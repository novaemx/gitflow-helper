# Gitflow Enforcement & Protection Strategy

## Incident Report

**Date**: April 19, 2026  
**Issue**: Packaging metadata files modified in feature branch  
**Severity**: Medium (Policy Violation)  
**Resolution**: Pre-commit hook installed + Protocol Documentation

## What Went Wrong

During previous interactions, the following files were modified in a feature branch (`feature/tui-fluid-activity-release-list`) when they should **only** be modified in `release/*` or `hotfix/*` branches:

```
✗ packaging/chocolatey/gitflow-helper.nuspec
✗ packaging/chocolatey/tools/chocolateyinstall.ps1
✗ packaging/homebrew/gitflow-helper.rb
✗ packaging/winget/*.yaml
```

These files contain version metadata that is tightly coupled to release management. Modifying them outside of a controlled release process violates the nvie gitflow model.

## Root Cause Analysis

### Technical Reasons

1. **No pre-flight `gitflow --json status` check** before starting modifications
2. **No file scope validation** before editing (what's allowed per branch type)
3. **Missing protective hooks** to prevent accidental commits
4. **Lack of whitelist validation** in the agent's decision tree

### Process Reasons

1. Scope creep: Feature work (TUI improvements) included unrelated packaging updates
2. Assumption of correctness: Assumed already in right branch without verification
3. No separation of concerns between feature delivery and release management

## Protection Mechanisms Deployed

### 1. Git Pre-Commit Hook ✅

**File**: `.git/hooks/pre-commit`

**Function**: Blocks commits of protected files outside `release/*` and `hotfix/*` branches

**Protected Files**:
- `packaging/`
- `VERSION`
- `CHANGELOG.md`
- `AGENTS.md`
- `RELEASE_NOTES.md`

**Behavior**: If a user attempts to commit these files on a feature/bugfix/develop/main branch, the hook will:
1. Display clear error message
2. List which protected files were staged
3. Explain correct workflow
4. Exit with status 1 (commit rejected)

### 2. Agent Protocol Documentation ✅

See: `/memories/session/gitflow-violation-analysis.md`

**Mandatory checks before ANY edit:**
- [ ] Run `gitflow --json status`
- [ ] Verify current branch is NOT `main` or `develop`
- [ ] Validate files to be edited against protection matrix
- [ ] Post-edit: Confirm diff doesn't touch protected files

### 3. File Protection Matrix

| File | feature/* | bugfix/* | release/* | hotfix/* |
|------|-----------|----------|-----------|----------|
| `packaging/*` | ❌ | ❌ | ✅ | ✅ |
| `VERSION` | ❌ | ❌ | ✅ | ✅ |
| `CHANGELOG.md` | ❌ | ❌ | ✅ | ⚠️ |
| `AGENTS.md` | ❌ | ❌ | ⚠️ | ⚠️ |

## Agent Instructions (Permanent)

For all future work in this repository:

1. **Before ANY file modification:**
   ```bash
   gitflow --json status
   ```

2. **Validate current branch** is NOT `main` or `develop`
   ```bash
   git branch --show-current | grep -v "^main$\|^develop$" || exit 1
   ```

3. **Whitelist validation** before editing:
   - Is this file allowed on this branch type?
   - Refer to File Protection Matrix above

4. **Post-commit verification:**
   ```bash
   git log -1 --name-only
   # Confirm touched files match expected scope
   ```

5. **If hook rejects commit:**
   - Read the error message carefully
   - Move changes to appropriate branch
   - Re-attempt commit
